    - Test 4: V100 diagnostic message contains "mutable static access requires synchronization"

  <action>

2. In `mir_converter.rs`, detect Place references to `static mut` items. In MIR, static accesses appear as `Place { local: _N, projection: [] }` where the local is a static item. Use `tcx.is_mutable_static(def_id)` or check `StaticKind::Static` with mutability. When detected, create a `StaticMutAccess` operation with the static's name and whether the access is a read or write (determined by context: LHS of assignment = write, otherwise read).

3. In `vcgen.rs`, in the unsafe operations processing loop, add a branch for `UnsafeOperation::StaticMutAccess`. Generate a `VcKind::DataRaceFreedom` VC. The VC assertion should encode: "there exists synchronization (Mutex/RwLock guard) around this access". For the SMT encoding, use a boolean flag `has_synchronization` that defaults to false. When the MIR analysis detects the access is within a MutexGuard scope (detect via local variable types in the same block), set the flag to true (making the VC trivially UNSAT = safe).

4. For Mutex/RwLock guard scope detection: use a simple heuristic -- scan the locals in scope at the static mut access point. If any local has type matching `MutexGuard` or `RwLockReadGuard` or `RwLockWriteGuard`, consider the access synchronized. Store this as a `synchronized: bool` field in StaticMutAccess or compute it during VCGen.

5. In `diagnostics.rs`, add V100 diagnostic: `"V100: DataRaceFreedom: mutable static access requires synchronization (Mutex/RwLock guard or atomic ordering)"`.
