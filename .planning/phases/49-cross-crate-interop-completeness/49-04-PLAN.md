- Returns `Some("From::from".to_string())` if it matches
- Returns None otherwise

Wire this into the existing None branch of `contract_db.get()` in both `generate_call_site_vcs` and `encode_callee_postcondition_assumptions`. When detected, use the From::from contract from contract_db for precondition checking and postcondition assumption.

Two changes needed:

**1. Add iterator adapter chain detection in vcgen.rs:**

The problem: when vcgen encounters a call to `Iterator::map` or `Iterator::filter`, it looks up the callee in contract_db by normalized name ("map", "filter"). These won't match "Iterator::map" / "Iterator::filter" keys. Even if they did match, there's no chain detection — each call is handled independently.

Add a helper function `detect_iterator_adapter_chain(func: &Function, call_site: &CallSiteInfo, contract_db: &ContractDatabase) -> Option<Contracts>` that:
1. Checks if the callee is a known iterator adapter (map, filter, take, etc.) by checking if `"Iterator::{callee_name}"` exists in contract_db
2. If yes, walks backward through the function's call sites (in the same path) to find the receiver — if the receiver (first arg) is the destination of another iterator adapter call, that's a chain
3. Collects all adapters in the chain as `(name, &StdlibContract)` pairs
4. Calls `compose_adapter_contracts(adapters)` from `crate::stdlib_contracts::iterator`
5. Returns `Some(composed_contracts)` if a chain of 2+ adapters found, None otherwise

For single adapters (no chain), still look up "Iterator::{name}" in contract_db and use that contract directly.

Wire this into `encode_callee_postcondition_assumptions`: after the existing contract_db.get() lookup fails for normalized names like "map"/"filter", try the iterator adapter detection. If composed contracts are returned, encode them as assumptions.

**Important:** Import `compose_adapter_contracts` from `crate::stdlib_contracts::iterator`. Also need access to the `StdlibContractRegistry` — but the current vcgen functions only receive `&ContractDatabase`. Two options:
- Option A: Pass `StdlibContractRegistry` alongside ContractDatabase (invasive change)
- Option B: When iterator contracts are merged into ContractDatabase via merge_into, the keys are "Iterator::filter", "Iterator::map" etc. Use contract_db.get("Iterator::filter") to check if it's an iterator adapter. Then for composition, construct the adapter list from contract_db entries. This avoids needing StdlibContractRegistry at all.

Use Option B. The compose_adapter_contracts function takes `&[(String, &StdlibContract)]` but we only have `FunctionSummary` from contract_db. Create a lightweight wrapper that builds `Contracts` from the chained summaries' postconditions directly, OR add a helper that works with `&ContractDatabase` instead of `&StdlibContractRegistry`. The simplest approach: create `compose_adapter_contracts_from_db(adapters: &[(String, &crate::contract_db::FunctionSummary)]) -> Contracts` in iterator.rs that mirrors compose_adapter_contracts but works with FunctionSummary.

Actually, the simplest approach is: in vcgen.rs, when detecting an iterator adapter call, construct the composed postconditions directly by collecting ensures from each adapter's FunctionSummary in the chain. This avoids adding new public API to iterator.rs. The composition logic (staged rewriting) can be replicated simply: concatenate all ensures from the chain, prefixed with stage annotations.

Use whichever approach is cleanest. The key requirement: chained adapter postconditions must NOT be BoolLit(true).
