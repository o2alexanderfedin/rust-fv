<objective>
Implement Drop scope-exit modeling with reverse-order VCs and Pin move-prevention enforcement.

Purpose: Drop ordering is critical for RAII correctness (file handles, locks, database connections). Pin is essential for async/self-referential types. Without verification, subtle drop-order bugs and invalid moves of pinned data go undetected.

Output: Drop scope-exit VCs in reverse declaration order, recursive field drop modeling, Drop+Copy diagnostic, Pin ghost state tracking, PinSafety VCs for non-Unpin types, and tests.
</objective>

  <name>Task 1: Drop scope-exit modeling with reverse-order VCs</name>
  <files>crates/analysis/src/vcgen.rs, crates/analysis/src/trait_analysis.rs, crates/driver/src/mir_converter.rs, crates/analysis/tests/drop_order_tests.rs</files>
  <behavior>
    - Test: Function with two Drop locals generates DropOrder VCs in reverse declaration order
    - Test: Struct with Drop fields has fields dropped before struct's own Drop
    - Test: Recursive drop — struct A contains struct B (both implement Drop) — B's drop VC appears before A's
    - Test: Type with Drop + Copy generates V140 diagnostic warning
    - Test: Non-Drop locals do NOT generate drop VCs at scope exit
    - Test: Drop::drop() with #[ensures] contract has postcondition checked at scope exit
  </behavior>
  <action>
    1. In trait_analysis.rs, add Drop/Unpin detection functions:
       - `pub fn has_drop_impl(ty: &Ty) -> bool` — checks if type implements Drop (by name matching against known Drop types: types with "Drop" trait bound, or struct names in a drop_impls registry)
       - `pub fn is_unpin(ty: &Ty) -> bool` — checks if type implements Unpin (primitives always Unpin, check trait bounds for user types)
       - `pub fn has_copy_impl(ty: &Ty) -> bool` — checks for Copy (for Drop+Copy diagnostic)

    2. In mir_converter.rs, populate `drop_locals`:
       - At function conversion, iterate locals in declaration order
       - For each local whose type has Drop impl, create `DropLocalInfo` with:
         - `local_name`: the SSA name
         - `ty`: the IR type
         - `has_drop`: true
         - `drop_order`: reverse of declaration index (last declared = first dropped)
       - Store in `func.drop_locals`

    3. In vcgen.rs, implement drop scope-exit VC generation:
       - At function return/exit (Return terminator), generate drop VCs:
       - Sort `func.drop_locals` by `drop_order` (reverse declaration)
       - For each drop-capable local:
         a. If the type has fields that also implement Drop, generate field drop VCs first (recursive)
         b. Generate the struct/type's own Drop VC
         c. If `Drop::drop()` has `#[ensures]` contracts, assert them as postconditions
       - Each VC uses VcKind::DropOrder with V140 diagnostic code

    4. In vcgen.rs, implement Drop+Copy diagnostic:
       - During function analysis, if any local's type has both Drop and Copy, emit a V140 warning diagnostic
       - This is a defensive check (Rust forbids it, but catches unsafe impls or proc macro issues)

    5. Write drop_order_tests.rs covering all behavior cases.
  </action>

<task type="auto" tdd="true">
  <name>Task 2: Pin move-prevention ghost state and PinSafety VCs</name>
  <files>crates/analysis/src/vcgen.rs, crates/driver/src/mir_converter.rs, crates/analysis/tests/pin_safety_tests.rs</files>
  <behavior>
    - Test: Pin::new_unchecked(&mut val) on non-Unpin type sets is_pinned ghost to true and generates PinSafety VC
    - Test: Moving a pinned non-Unpin value generates MemorySafety VC asserting !is_pinned (Z3 SAT = violation)
    - Test: Pin::new() on Unpin type sets is_pinned but move is allowed (Unpin bypasses)
    - Test: Pinned struct field projection produces pinned reference to field (structural pinning)
    - Test: Pin::new_unchecked without #[unsafe_ensures] generates PinSafety VC requiring user contract
    - Test: Non-pinned value move generates no Pin-related VCs (regression)
  </behavior>
  <action>
    1. In mir_converter.rs, detect Pin types and operations:
       - Detect `Pin<P>` wrapper type by name matching (ADT path contains "pin::Pin" or "std::pin::Pin")
       - Detect `Pin::new()` and `Pin::new_unchecked()` call sites
       - For each local wrapped in Pin, create `PinGhostState { local_name, inner_ty }`
       - Store in `func.pin_ghost_states`

    2. In vcgen.rs, implement Pin ghost state SMT preamble:
       - For each `PinGhostState`, declare ghost variable `{local}_is_pinned: Bool` initialized to false
       - At `Pin::new()` call site: set `{local}_is_pinned = true`
       - At `Pin::new_unchecked()` call site:
         a. Set `{local}_is_pinned = true`
         b. If inner type is NOT Unpin, generate `PinSafety` VC (V150) — requires caller to provide `#[unsafe_ensures]` to discharge

    3. In vcgen.rs, implement move-prevention VCs:
       - At any MIR `Move` operation on a local that has pin ghost state:
         a. Check if `is_pinned` is true AND type is NOT Unpin
         b. If so, generate `MemorySafety` VC: `assert(!{local}_is_pinned || is_unpin({local}_ty))`
         c. For Unpin types, this VC is trivially UNSAT (safe)
         d. For non-Unpin types, if pinned, VC is SAT (violation detected)

    4. Implement structural pinning for field projections:
       - When accessing a field of a pinned struct (`pinned_struct.field`):
         a. If the struct is pinned, the field reference inherits `is_pinned` status
         b. Create a derived `PinGhostState` for the field projection
         c. Field moves from pinned structs also check `is_pinned`

    5. Use trait_analysis.rs `is_unpin()` (from Task 1) for Unpin detection.

    6. Write pin_safety_tests.rs covering all behavior cases.
  </action>
